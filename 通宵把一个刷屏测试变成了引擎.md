# 通宵把一个刷屏测试变成了引擎——XBTI 的技术拆解

昨晚刷到 SBTI 测试的时候，我的第一反应不是去测，而是去看源码。

如果你这两天刷过朋友圈或者微博，你大概率见过它——一个叫 [SBTI](https://xbti.lovstudio.ai/c/sbti) 的「傻逼人格测试」，B站UP主[蛆肉儿串儿](https://space.bilibili.com/417038183)做的，30 道题告诉你是 27 种离谱人格中的哪一种。DEAD（已死者）、IMSB（自我攻击者）、MALO（吗喽）、ATM-er（送钱者）……每个人格都「全中国最稀有」，每个结果都让人截图发群。

它上线几个小时就炸了服务器，[冲上了微博热搜](https://www.sina.cn/news/detail/5285843166036354.html)，小红书、朋友圈、抖音同步刷屏。用户量直接打到 4 万+，逼得大家截图「云测」。

但让我真正停下来的不是流量数据，而是它的匹配算法。一个恶搞测试，用的是 **15 维向量空间上的曼哈顿距离匹配**。这比绝大多数正经 MBTI 测试的技术含量都高。

于是我通宵干了一件事：把 SBTI 从一个单页应用，重构成了一个通用的人格测试引擎——[XBTI](https://github.com/lovstudio/XBTI)。任何人只要填一个数据目录，就能生成自己主题的 BTI 测试，零代码改动。

这篇文章拆解的不是「SBTI 有多火」，而是：**一个爆款玩具的底层，藏着什么值得认真对待的工程问题？**

## 先看算法：为什么一个恶搞测试比正经 MBTI 更硬核

大多数网上的 MBTI 测试，技术实现是这样的：4 个维度（E/I、S/N、T/F、J/P），每个维度统计倾向百分比，四个字母拼出来就是结果。本质上是 4 次独立的二选一，数学复杂度约等于零。

SBTI 不一样。它的底层是一个 **15 维评估系统**。5 个模型（自我、情绪、态度、行动、社交），每个模型 3 个子维度，共 15 维。30 道题，每维度 2 题，每题 3 个选项（1/2/3 分）。

有意思的地方在分数到人格的映射。它没有用简单的阈值切分，而是走了一条更优雅的路：

**第一步，离散化。** 每个维度的原始分（2-6 分）映射为三级：L（≤3）、M（=4）、H（≥5）。这一步看似粗暴，实际上是在做降维——把连续空间压缩到 3^15 = 14,348,907 个离散状态。

**第二步，向量化。** L/M/H 转为数值 1/2/3，构建一个 15 维用户向量。每个人格类型也有一个 15 维模式向量，用类似 `HHH-HMH-MHH-HHH-MHM` 的 pattern 字符串表示，人眼可读，版本可控。

**第三步，匹配。** 计算用户向量与所有候选人格的[曼哈顿距离](https://en.wikipedia.org/wiki/Taxicab_geometry)（L1 范数）。距离最小的就是你的人格。相似度公式：`(1 - distance/30) × 100%`，距离 0 就是 100% 匹配。

为什么用曼哈顿距离而不是欧几里得距离？因为数据是离散等级（L/M/H），不是连续浮点数。曼哈顿距离对这类有序分类数据的区分度更好——它是线性的，不会因为某个维度偏差大就指数级放大影响。一个维度从 L 到 H 的「代价」就是 2，不多不少，各维度平等。

排序规则也有讲究：先按总距离升序，距离相同按「完全命中维度数」降序，再按相似度百分比降序。三级排序保证了结果的稳定性和可解释性。

还有两个防护机制值得一提：

1. **兜底人格**：如果最佳匹配的相似度低于 60%，强制分配到一个 fallback 类型，附带「系统强制分配」的提示。宁可给一个不太准的结果，也不给一个完全不靠谱的。
2. **隐藏人格**：如果答题时在酒量问题上选了「白酒保温杯」，直接跳过整个匹配算法，返回 DRUNK（酒鬼）类型——原作者做这个测试的初衷就是想劝朋友少喝酒。

88 行 JavaScript，完成了一个有效的 15 维人格匹配系统。说实话，大多数正经心理测试的代码都没这个干净。

## 从玩具到引擎：三次变形

把 SBTI 变成 XBTI，不是一次重构，是三次。

### 第一次：从 HTML 到 React

原版 SBTI 是一个[单文件 HTML](https://github.com/UnluckyNinja/SBTI-test)。所有题目、人格、匹配逻辑全部写在一个文件里。这种形态有个优点：部署极简（甩一个文件到任何静态服务器就行），但完全不可扩展。

第一步是拆成 React 组件。6 个组件，各司其职：

```
App.jsx          → 路由分发
HomePage.jsx     → 首页（选择哪个 BTI）
TestFlow.jsx     → 状态机（intro → test → result）
CaseIntro.jsx    → 变体介绍 + 人格画廊
TestScreen.jsx   → 答题（含题目洗牌和条件注入）
ResultScreen.jsx → 结果展示 + 15 维详解
```

这里有个细节：`TestFlow` 用 React 的 `key` 机制来重置状态。切换 BTI 变体时，`key={caseId}` 变了，React 会卸载再重新挂载整个组件树，所有内部状态自动清零。比手动 `setState(initialState)` 干净得多，也不容易漏。

### 第二次：从硬编码到数据驱动

这是最关键的一步。原版代码里人格名字、题目文本、维度定义都散落在组件逻辑里。要支持多主题，必须把**数据和逻辑彻底分离**。

最终的 case 数据结构长这样——每个 BTI 变体是 `cases/` 下的一个目录，4 个文件：

| 文件 | 职责 | 内容举例 |
|------|------|---------|
| `index.js` | 元信息 + 统一导出 | id、名称、作者、简介 |
| `dimensions.js` | 15 维度定义 | 维度名、所属模型、L/M/H 解释 |
| `questions.js` | 题目池 | 30 道常规题 + 特殊题 + 触发条件 |
| `types.js` | 人格库 | 代号、中文名、描述、15维模式 pattern |

所有组件只接收 `caseData` 对象作为 prop，不关心具体内容是恶搞人格还是猫猫人格。评分引擎 `scoring.js` 完全通用——它只认向量和距离，不认主题。

这意味着加一个新的 BTI 变体，**零行代码改动**。写 4 个数据文件，在 `registry.js` 里加一行注册，完事。

### 第三次：从单入口到多 case 路由

最后一步是路由。一个引擎要同时跑多个 BTI 变体，URL 结构必须反映这一点：

```
/           → 首页（所有变体列表）
/c/sbti     → SBTI 介绍页
/c/sbti/test → SBTI 答题
/c/cbti     → CBTI 介绍页
/c/cbti/test → CBTI 答题
```

路由逻辑写在一个 22 行的自定义 hook `useHashRoute.js` 里（名字有误导，实际用的是 History API，不是 hash）。`App.jsx` 解析 URL 段，查注册表拿到 case 数据，传给 `TestFlow`。整个路由层加起来不到 50 行。

没有用 React Router。对于这个规模的应用，引入一个路由库是过度工程。URL 结构简单固定，手写路由反而更可控。

## 题目洗牌和条件注入：细节里的产品思维

`TestScreen` 的题目处理逻辑值得单独拿出来说，因为它反映了一种工程上的克制。

30 道常规题每次进入测试时随机打乱顺序。这不是为了防作弊——恶搞测试没有作弊动机——而是为了**降低社交比较时的记忆偏差**。你和朋友讨论「第 5 题你选了什么」时，其实你俩看到的不是同一道题。这减少了「被剧透答案」的体验损耗。

特殊题目的注入更有意思。第一道特殊题随机插入到题目序列中的某个位置。如果用户在饮酒相关问题上选了特定答案，第二道特殊题才会被动态注入到序列中——而如果用户改了主意，第二道特殊题会自动移除，连答案都一起清掉。

这段逻辑大概 20 行代码，但它实现了一个完整的条件分支问卷，没有引入任何问卷框架或状态管理库。

## 数据层设计：SBTI vs CBTI，同一引擎两张皮

目前引擎上跑着两个变体：SBTI（恶搞人格，27 种）和 [CBTI](https://xbti.lovstudio.ai/c/cbti)（猫猫人格，18 种）。

它们的维度体系完全不同：

| | SBTI | CBTI |
|--|------|------|
| 模型 1 | 自我模型（自尊、认知、接纳） | 社交模型（亲人度、领地意识、社牛指数） |
| 模型 2 | 情绪模型（依赖、安全感、波动） | 作息模型（夜行指数、活力值、睡眠深度） |
| 模型 3 | 态度模型（规则、秩序、道德） | 狩猎模型（捕猎欲、好奇心、破坏力） |
| 模型 4 | 行动模型（决断、执行、冒险） | 情绪模型（傲娇值、戏精指数、记仇值） |
| 模型 5 | 社交模型（人际、合群、表达） | 生存模型（挑食度、适应力、颜值管理） |

但引擎完全不关心这些语义差异。`scoring.js` 看到的只是「15 个维度，每个有 L/M/H 三级，跟 N 个 pattern 做距离匹配」。维度叫什么、描述怎么写、有几种人格，全部是数据层的事。

这个分离做到位了，就意味着你可以用同一套代码做「程序员人格测试」「奶茶人格测试」「甲方人格测试」——只要你能定义出 15 个维度和一批人格 pattern。

## 为什么是 15 维？

这是我重构过程中一直在想的问题。MBTI 是 4 维（准确说是 4 对二元对立），大五人格是 5 维，SBTI 选了 15 维。

15 = 5 × 3 的结构很巧妙。5 个大模型对应人格的 5 个面向（这跟大五人格的 5 因素有隐约的呼应），每个模型下 3 个子维度提供精细度。

更重要的是数学上的效果：15 维 × 3 级 = 3^15 ≈ 1430 万种理论组合。而 SBTI 只定义了 25 种正常人格（加上隐藏和兜底共 27 种），覆盖率不到万分之二。这意味着**大部分用户不会得到完美匹配，而是得到「最接近的」人格**——这反而是一个正确的设计。

因为人格测试的体验核心不是精确分类，是**共鸣**。给你一个 85% 匹配的结果，你会自动脑补那 15% 的偏差，觉得「说得挺准的」。给你一个 100% 匹配但描述平淡的结果，你反而觉得不准。曼哈顿距离 + 稀疏人格库的组合，天然地制造了这种「差一点的共鸣感」。

## 1800 行代码的克制

整个项目大约 1800 行代码（源码 + 数据）。没有状态管理库，没有 CSS 框架，没有路由库，没有测试框架。React 19 + Vite 8 + 一个自定义 hook，结束。

这不是因为时间不够所以没加。这是因为**每多引入一个依赖，就多一个需要维护、需要理解、需要升级的东西**，而这个项目的核心价值在数据层，不在工程复杂度。

评分逻辑 88 行。路由 22 行。状态管理全部是 `useState`。CSS 用的是原生自定义属性。整个项目 `node_modules` 之外的部分，任何会写 React 的人 30 分钟能读完。

对于一个开源引擎来说，这比功能丰富重要得多。因为用它的人需要理解它，而不只是使用它。

## 从 SBTI 到 XBTI：一个通宵学到的东西

通宵做这件事最大的收获不是代码本身，而是对「泛化」这个词的重新理解。

很多时候我们做泛化，是先设想「未来可能需要什么」，然后提前抽象。这几乎总是过度工程。XBTI 的泛化路径是反过来的：**先有了 SBTI 和 CBTI 两个具体实例，然后问「它们的共同结构是什么」**。

答案很清晰：15 维评估 + 三级量化 + 曼哈顿距离匹配。这个骨架是稳定的，变化的只是维度的语义、题目的内容、人格的定义。所以抽象层应该画在这条线上，而不是更高或更低。

另一个感触是：**算法的简单性和产品的有趣性不矛盾**。曼哈顿距离是最朴素的度量之一，L/M/H 三级量化是最粗暴的离散化方式，但组合起来就是能给出让人觉得「有点准」的结果。不需要机器学习，不需要大模型，不需要用户画像。30 道题，88 行匹配逻辑，够了。

下次有人跟你说「这个功能需要接大模型」的时候，先问一句：**曼哈顿距离够不够用？**
